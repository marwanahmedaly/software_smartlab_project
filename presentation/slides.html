Segment 1: Project Overview

The Problem

Speaker: Marwan Aly · Time: 90 seconds

🎯 Problem Addressed

  • Computer labs managed reactively — equipment failures discovered only after users complain
  • Repair histories scattered in spreadsheets and emails
  • Technicians lack structured queues; administrators lack visibility
  • Result: Preventable downtime disrupting teaching (especially during exam periods)

👥 Target Users

  • Lab Administrators: Need visibility into hardware health and budget planning data
  • Maintenance Technicians: Need prioritised repair queue and intelligent diagnostic guidance
  • Regular Users (students/staff): Need frictionless way to report faults

🎯 Purpose

  • Reduce device downtime, lower IT overhead, improve learning environment
  • Aligns with UN SDG 4 (Quality Education)

📊 Data Flow

User clicks "Report Fault" → Frontend POST /api/issues → Backend validates JWT → Saves to SQLite → Notifies technicians → Technician opens AI Diagnostics → Backend enriches prompt with device specs + history → Calls Llama 3.2 via OpenRouter → Returns Arabic diagnosis in < 10 seconds → Technician updates status → Admin views reports

Marwan's Reflection

Speaker: Marwan Aly · Time: 5 minutes

🛠️ Specific Task: Building JWT Authentication and REST API

Tool: Claude Code + GitHub Copilot
Why AI? To accelerate boilerplate scaffolding so I could focus on security logic and business rules
Input: "Create Express server with JWT middleware, SQLite setup, and REST routes with role-based access."
Output: Route skeletons, auth middleware pattern, SQLite table definitions
How I Revised It: AI generated middleware with hardcoded JWT_SECRET = 'secret' — revised to process.env.JWT_SECRET. AI schema missing FOREIGN KEY constraints — added manually to enforce referential integrity.

💭 Reflection

Useful: Rapid prototyping — auth system working in hours instead of days
Challenging: Security-critical code needed manual review; AI doesn't understand timing attacks or secret rotation
Limitation: AI suggested MongoDB initially; some SQL syntax was generic
Future: Use AI for scaffolding but personally review all auth and security code before committing

Segment 3: Individual Reflection

Aya's Reflection

Speaker: Aya Abdelmonem · Time: 5 minutes

📊 Specific Task: Reports Dashboard, Analytics SQL, and Project Documentation

Tool: Claude Code + GitHub Copilot
Why AI? Documentation and SQL aggregations are structured but tedious — AI excels at boilerplate
Input: "Write SQLite queries for maintenance summary with date filtering, including average repair time and most-faulted device."
Output: Chart.js bar chart configuration, SQL aggregation queries, README.md and PLAN.md drafts
How I Revised It: AI query: BETWEEN '2024-01-01' AND '2024-01-31' — revised to >= '2024-01-01' AND < '2024-02-01' because SQLite BETWEEN is inclusive and misses time-of-day boundaries. AI README had placeholder .env values — replaced with actual .env.example content.

💭 Reflection

Useful: Documentation scaffolding saved hours of formatting and structure planning
Challenging: AI-generated SQL had subtle date boundary bugs — queries returned wrong aggregates until tested against real seed data
Limitation: AI suggested complex nested JOINs that performed poorly on SQLite — simplified to targeted queries. Documentation included features not yet built.
Future: Use AI for structure and drafts, but always run queries against real data and benchmark performance before accepting. For docs: generate skeleton, then fill in only implemented features.
