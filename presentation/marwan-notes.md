**Opening:**

## 📝 Slide 3: System Architecture (60 sec)

**Opening:**
> "Before we show the system, let me explain how it fits together."

**Architecture:**
> "SmartLab is a **four-tier web application**. The frontend is vanilla HTML, CSS, and JavaScript — zero framework overhead, runs on any browser. It communicates via HTTP REST to our **Node.js/Express backend**, which handles JWT authentication and business logic."

**Data Layer:**
> "For persistence, we chose **SQLite** — zero-config, file-based, perfect for a single institution. For AI diagnostics, we integrated **Llama 3.2 via OpenRouter** — an on-premise alternative to cloud LLMs that keeps data within the institution."

**Data Flow:**
> "Here's the flow: a user clicks 'Report Fault' — the frontend sends a POST request to `/api/issues`. The backend validates the JWT, saves to SQLite, and notifies technicians. When a technician opens AI Diagnostics, the backend enriches the prompt with device specs and fault history, calls Llama 3.2, and returns an Arabic diagnosis in under 10 seconds."

## 📝 Slide 8: AI Usage — Backend Implementation (90 sec)

**Opening:**
> "For the backend, I used Claude Code and GitHub Copilot to scaffold the Express server, JWT authentication, and SQLite schema."

**Prompt:**
> "My prompt was: *'Create Express server with JWT middleware, SQLite setup, and REST API routes for devices and issues with role-based access control.'*"

**Outcome:**
> "The AI generated route skeletons, auth middleware patterns, and a 5-table SQLite schema."

**Revision — Critical:**
> "But I had to revise two critical things. First, the AI generated JWT middleware with a **hardcoded secret** — `JWT_SECRET = 'secret'`. I revised this to use `process.env.JWT_SECRET` for security. Second, the AI schema was missing **foreign key constraints** — I added them manually to enforce referential integrity."

**Team Decision:**
> "Our team also evaluated MongoDB versus SQLite. We chose SQLite because it's zero-config, file-based, and perfect for a single-institution deployment with no external dependencies."

**Limitation:**
> "The limitation? AI initially suggested MongoDB with Mongoose — we had to override that. Some SQL syntax was also generic and needed refinement for the better-sqlite3 dialect."

## 📝 Slide 13–14: Individual Reflection (5 min)

**Opening:**
> "My specific task was building the JWT authentication and REST API using AI."

**Tool & Purpose:**
> "I used **Claude Code and GitHub Copilot**. I chose AI because it accelerates boilerplate scaffolding, allowing me to focus on security logic and business rules instead of writing route handlers from scratch."

**Input & Output:**
> "My prompt was: *'Create Express server with JWT middleware, SQLite setup, and REST routes with role-based access.'* The AI produced route skeletons, an auth middleware pattern, and SQLite table definitions."

**How I Revised It — KEY POINT:**
> "Here's what I changed: the AI generated middleware with `JWT_SECRET = 'secret'` hardcoded. **I revised this to use environment variables** — `process.env.JWT_SECRET`. The AI schema also missed foreign key constraints. **I added them manually** to enforce referential integrity between devices, issues, and users."

**What Was Useful:**
> "The AI was incredibly useful for rapid prototyping — I had a working auth system in hours instead of days."

**What Was Challenging:**
> "The challenge was that security-critical code needs manual review. AI doesn't understand timing attacks, secret rotation, or the principle of least privilege."

**Limitations:**
> "AI suggested MongoDB initially, which would have added unnecessary complexity. Some SQL syntax was also too generic."

**Future:**
> "Going forward, I'll use AI for scaffolding but **personally review all authentication and security code** before committing. AI is an assistant, not a security auditor."
